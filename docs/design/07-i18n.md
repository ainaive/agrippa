# 07 — i18n Strategy (en / zh-CN)

> Status: draft for review · Last updated: 2026-07-17

One library — **i18next** — serves the SPA (react-i18next), the API (error messages), and the worker (emails/notifications). Two locales in M1: `en`, `zh-CN`. Rationale vs. compile-time alternatives (paraglide): DB-stored metadata and runtime locale switching need a runtime i18n layer anyway ([ADR-0008](../adr/0008-i18n-jsonb-columns.md) covers the data side).

## The Three Kinds of Localizable Text

| Kind | Lives in | Mechanism |
|---|---|---|
| **UI strings** (labels, buttons, empty states) | `packages/i18n/locales/{en,zh-CN}/*.json` | react-i18next, namespaces: `common`, `scenarios`, `runs`, `admin`, `errors` |
| **Backend messages** (API errors, emails) | same package, `errors` / `emails` namespaces | server-side i18next instance; locale middleware picks the language; error `code` stays stable |
| **DB-stored metadata** (scenario/task-type/template/faber names, labels in template inputs) | `*_i18n jsonb {"en","zh-CN"}` columns; same shape in template YAML | shared `pickLocale(obj, locale)` helper in `@agrippa/core` |

`pickLocale` fallback chain: requested locale → `en` → first available key. Used identically by API serializers and SPA components, so fallback behavior can't diverge.

## Locale Detection & Switching

Priority: `?lang=` query → authenticated user's `users.locale` → `Accept-Language` → `en`.

The SPA switcher updates i18next in place (instant, no reload), persists to localStorage (pre-auth) and `PATCH /me` (post-auth). Emails/notifications use the recipient's stored locale, not the actor's.

## Authoring Rules

- Keys are semantic (`runs.status.waiting_approval`), never English-sentence keys.
- Interpolation via i18next placeholders (`{{count}}`, plural forms) — zh-CN has no plural forms; rely on i18next plural handling per language.
- Template YAML must provide both `en` and `zh-CN` for every `name`/`label`/`description` — the template compiler **fails validation** on missing locales (builtin templates therefore can't ship half-translated).
- Dates/numbers via `Intl` with the active locale; costs always shown as USD with locale-appropriate formatting.

## Enforcement in CI

A `bun test` check in `packages/i18n` asserts **key parity** between `en` and `zh-CN` across all namespaces (missing or orphaned keys fail CI). Combined with the compiler rule above, untranslated strings cannot land silently.
