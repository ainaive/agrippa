# ADR-0013: Per-Project Provider Credentials and Single-Provider Slot Resolution

- Status: accepted · Date: 2026-07-24

## Context

Provider authentication was worker-env only: `ANTHROPIC_*` / `OPENAI_*` variables set on the worker process, allow-listed into every executor subprocess. That makes provider choice a deployment property — a project cannot run Aliyun Bailian (DashScope) Qwen models without repointing the entire deployment's `ANTHROPIC_BASE_URL`, and two projects cannot bill to different keys. Bailian also has no legitimate deployment-wide default: its endpoints only work with a customer's own key, so a "worker env fallback" for it is a misconfiguration, not a feature.

Two structural facts shaped the design. First, an executor process has exactly one base URL: a single SDK query serves the step's model *and* every subagent model, so any resolution that mixes providers inside one agent slot cannot execute. Second, slots used to resolve **every** template role regardless of which roles their steps reference, which would have forced every provider a slot can use to cover roles the slot never runs.

## Decision

1. **Project-scoped credential rows over org- or deployment-scoped config.** A `provider_credentials` table holds one row per (project, provider) with an optional base-URL override; the key lives encrypted in the existing `secrets` table (new kind `provider_api_key`), following the repo-connection/MCP secretRef precedent: write-only API, masked reads (`hasCredential`), audit on every mutation, decrypt only in the worker. Rotation updates the ciphertext in place so the reference stays stable; DELETE removes the secret in the same transaction (no orphaned key material). Clearing a key *is* deletion — a keyless credential row is meaningless, so the API has no `apiKey: null` state.
2. **A static provider catalog in `@agrippa/core`.** `PROVIDER_CATALOG` mirrors `EXECUTOR_CATALOG`: per-provider default base URLs keyed by **wire protocol** (`anthropic` / `openai`), because one provider can serve both executor families (Bailian exposes an Anthropic-compatible endpoint for claude-agent-sdk and an OpenAI-compatible mode for codex-cli), plus an auth policy — `"env"` (worker env is a legitimate fallback: anthropic, openai) or `"project"` (a project credential is required: dashscope). Unknown providers default to `"env"` with no base URL, so the catalog never blocks a custom registry entry.
3. **Project credential wins over worker env.** The credential travels as an optional, executor-agnostic `providerAuth` field on `StepExecutionRequest`; each executor maps it onto its own auth family via `overlayProviderAuth`, which **deletes the protocol's entire auth-var family** before setting the mapped variables. Deleting the family is what makes precedence unambiguous — SDKs prefer some variables over others (e.g. an OAuth token over a bearer token), so merging instead of replacing would let a leftover worker-env variable silently outrank the project key.
4. **Single-provider slot resolution, scoped to the roles a slot uses.** Each slot resolves only the model roles its steps (and their subagents) reference. Provider-constrained slots must satisfy *all* of those roles from one provider; candidates lacking a required project credential are excluded, and the ranking is deterministic: credentialed provider first, then lowest total input cost, then provider id. When the only blocker is a missing credential the submit fails with `provider_credential_required` instead of a misleading `model_unresolvable`. The `"*"` provider set (fake executor, demo mode, uncataloged custom executors) keeps the legacy mixed resolution with no credential gating so token-free demos are unchanged.
5. **Codex via `-c` overrides and a per-run CODEX_HOME.** When a credential carries an effective base URL, the codex adapter appends a synthesized `model_providers` entry (`base_url`, `env_key=OPENAI_API_KEY`, `wire_api=chat`) as `-c` arguments — they survive `--ignore-user-config`, which stays on. With a project credential the run also gets its own `CODEX_HOME` under the OS tmpdir so an ambient `auth.json` cannot outrank the project key; per **run**, not per step, because resume sessions live under `CODEX_HOME`. The worker now registers codex on a successful CLI probe alone — env auth at boot is no longer a precondition since auth can arrive per step.
6. **Submit-time gating uses only what the API can see.** The API checks `provider_credentials` rows plus the catalog auth policy; it does not know the worker's env. For `auth: "env"` providers it assumes the deployment fallback exists — an upgrade path is to have workers advertise their env-auth providers through the existing `executor_registrations` heartbeat and consult that instead of assuming.

## Alternatives considered

- **Deployment-level DashScope env vars.** Works for a single-tenant install but makes provider choice global, prevents per-project billing, and forces one of anthropic/dashscope to be unusable at a time (one process-wide `ANTHROPIC_BASE_URL`).
- **Freezing the key or base URL into `runs.model_resolution` at submit.** Would keep resolution self-contained, but puts secret-adjacent data in the run record and breaks rotation mid-run. Resolving lazily in the engine (mirroring MCP auth materialization) keeps secrets out of run rows and lets a rotated key apply on the next step.
- **Mixed-provider slots with per-subagent base URLs.** Not implementable: the executor subprocess env is process-wide, and neither SDK accepts per-model endpoints inside one query.
- **Org-level credential sharing.** Deferred; the project scope matches the existing grants/quota trust boundary, and an org tier can be layered on later without schema changes to this table.

## Consequences

- A project with a dashscope credential and granted Qwen models runs entirely on Bailian — including through codex — while sibling projects keep the worker's Anthropic/OpenAI keys; the decrypted key is registered with the run's redactor at materialization, before any request can echo it into an event payload.
- Credentialed-first ranking means adding a provider credential can switch which provider a slot resolves to on the next submit — that is the intended meaning of "the project chose this provider," and it is visible in the frozen resolution on the run detail page.
- Per-slot role scoping shrinks `runs.model_resolution` to the roles each slot actually uses; readers already tolerate missing roles because `modelFor` only looks up roles the slot's steps reference.
- Project deletion cascades the credential row but strands the org-scoped secret — the same pre-existing behavior as repo connections; acceptable because projects are archived, not deleted.
- Qwen seed prices are entry-tier list rates; Bailian tiers pricing by input length, so operators should verify rates (and the region endpoint) before relying on budget enforcement.

## Amendment (2026-07-24) — review-round corrections

The first external review of this feature invalidated one decision and hardened three others:

1. **dashscope is claude-only for now; decision 5's codex wiring is staged out.** Codex CLI ≥ 0.122 rejects `wire_api = "chat"` outright, and Bailian's OpenAI-compatible mode is chat-completions (its Responses API rollout is per-model and does not yet cover the seeded Qwen models). `codex-cli` therefore serves `openai` only again, the dashscope catalog entry keeps only its Anthropic-protocol endpoint, and the synthesized `model_providers` block no longer sets `wire_api` — the CLI's `responses` default is correct for explicit `openai` proxy overrides, which remain supported. Re-enable path: when Bailian's Responses API covers the Qwen models we seed, add `dashscope` back to codex's providers (no `wire_api` argument needed). The fixture-based executor tests validate argv/env mapping only, not CLI config semantics — an accepted limitation; the incompatible config is removed rather than integration-tested.
2. **A credential's baseUrl is validated as a key-egress target.** The worker sends the decrypted key to that URL, so a syntactically-valid-URL check was an exfiltration/SSRF channel (one project admin could PATCH the baseUrl and redirect a key another admin entered). `validateProviderBaseUrl` enforces: https only, no userinfo/query/fragment, public DNS hostnames only (IP literals, `localhost`, and dotless internal names rejected — WHATWG parsing canonicalizes numeric IPv4 forms first), and per-provider host families via the catalog's new `baseUrlHosts` (dashscope pins `.aliyuncs.com`). Resolve-time DNS/rebinding checks are deliberately out of scope: deployments needing an internal proxy configure worker env, which is operator-owned. Complementing this, `effectiveBaseUrl` refuses to apply a row override to a wire protocol the provider is not known to serve, so a single override can never leak across protocols.
3. **Credentials materialize per step, not per run.** The engine's per-run memo contradicted this ADR's own consequence ("a rotated key applies on the next step"); the lookup now runs fresh for every step request. Cost is one indexed row + AES-GCM per agent step; superseded keys stay in the redactor's set.
4. **Retry re-asserts the credential gate.** Retries copy the frozen resolution verbatim, so a credential deleted between runs used to surface as an auth failure mid-run; `assertResolutionCredentialed` now re-checks every `auth: "project"` provider in the frozen resolution (both shapes; demo/uncataloged slots skipped) before the new run row exists. Credential PATCH also became a single transaction — a partially applied rotate with no audit row would break the every-mutation-is-audited invariant.

## Amendment 2 (2026-07-24) — second review round

1. **Endpoint and key travel together.** A baseUrl-only PATCH could still redirect a write-only key another admin entered (the round-1 host rules only constrained *where*). Changing `baseUrl` to a new endpoint now requires re-entering `apiKey` in the same request — an attacker can only point a key they already know at a new host; clearing to the trusted catalog default (`baseUrl: null`) needs no key. Residual (accepted, audited): an admin who plants their own key + endpoint and waits for someone to rotate the key in place — endpoint changes are audit-logged (`baseUrlChanged`) and the effective endpoint is always visible on the credential row. An operator-managed allowlist for `anthropic`/`openai` overrides remains future work.
2. **Dashscope host pins exclude customer-controlled Aliyun neighbors.** `.aliyuncs.com` admitted OSS bucket domains (`bucket.oss-cn-*.aliyuncs.com`), which are customer-owned and can log request headers. `baseUrlHosts` entries are now exact hosts or explicit suffixes: `dashscope.aliyuncs.com`, `dashscope-intl.aliyuncs.com`, `.maas.aliyuncs.com`.
3. **The worker rejects base-URL hosts that resolve into private space.** Syntactic checks only catch IP literals; a DNS name resolving to loopback/private/link-local (rebinding, internal SSRF) is now rejected at credential materialization (`assertPublicHost`, per step), failing the run as `base_url_invalid` via the typed `ProviderCredentialError`. Rebind-after-check (TOCTOU) remains possible — this raises the bar; it is not a network boundary.
4. **Mid-run loss of a required credential fails the run actionably.** Per-step materialization (amendment 1 §3) returned undefined on deletion, silently falling back to env auth that cannot exist for an `auth: "project"` provider. On credential-gated executors the engine now throws `RunFailure("provider_credential_required")` before executor invocation — the same code the submit and retry gates use. Demo/`fake` and uncataloged executors stay exempt (`isCredentialGatedExecutor`), since `"*"` resolution legitimately picks such models token-free.
5. **Keyless workers decline runs they cannot authenticate.** Probe-only registration (decision 5) let a codex worker with no OpenAI auth claim env-fallback runs. Executors now advertise `envAuthProviders` (captured from their env at construction); before the queued→running transition the engine checks each slot's env-policy providers against that list and the project's credentials and, if neither covers one, throws `ExecutorUnavailableError` — the existing heterogeneous-fleet decline: the worker defers with a visible `run.deferred` reason and a capable worker picks the run up. An all-keyless fleet leaves the run queued with explanatory timeline events rather than burning it. Fleet-wide submit-time env-auth advertisement (heartbeat column) remains the future refinement of amendment 1's upgrade path.
