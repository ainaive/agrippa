# 核心概念

**项目（Project）** —— 协作单元，也是一切事物的边界：成员、资源授权、预算/配额、代码仓库与计费归属。用户可以以不同角色加入多个项目：**管理员**（管理成员、授权、仓库、配额）、**成员**（提交任务、决定审批、取消执行）、**访客**（只读）。

**场景与任务类型** —— 任务目录按工作场景组织（项目管理、软件研发、测试验证）。*任务类型*是一件可以具体提交的事——如「缺陷定位与修复」「测试计划」——它把场景、编排模板与默认硅基人绑定在一起。

**任务与执行（Task / Run）** —— *任务*是你的一次提交：任务类型 + 标题 + 参数。执行任务产生一次*执行*；对已结束的任务重试会产生第 2、3 次执行。每次执行都固定在提交那一刻的模板版本上，之后的模板变更不会影响它。

**步骤（Step）** —— 执行内部的原子单元：一次智能体调用（或一次系统动作，如检出代码仓库）。步骤构成执行时间线，也是平台重试与崩溃恢复的基本单位。

**硅基人（Faber，复数 Fabri）** —— 平台预置的智能体：一套人设与系统提示词，任务在其之下执行。内置四位：领航者 Navigator（项目管理）、铸造者 Forge（软件研发）、哨卫 Sentinel（测试验证）、裁决者 Arbiter（代码评审）。

**代理位与执行器** —— 较新的模板会声明具名的*代理位*（如**实现者**与**评审者**），每个代理位由一位硅基人人设加一个*执行器*（真正驱动智能体的引擎——Claude Code 或 OpenAI Codex）组成。可覆盖的代理位允许你在提交时更换人设或引擎；两个角色在设计上可以互换——前提是所选引擎具备条件：部署中该引擎可用，且项目已被授权其提供商的模型，因此提交表单只会给出真正能运行的组合。

**编排模板** —— 执行遵循的带版本声明式配方：输入参数（自动生成提交表单）、阶段与步骤、每步使用的技能/MCP 服务/子智能体、模型选择规则、人工检查点（审批、问题表单、评审关卡）、预算与产出物契约。已发布版本不可变；修改会产生下一个版本。

**模型角色与档位** —— 模板从不指名具体模型，而是声明角色（如 `planning`、`coding`、`fast`）并映射到档位（**strong / balanced / fast**）；提交时平台把每个角色解析为该档位中已授权的最便宜模型。更换项目的模型阵容无需改动任何模板。

**产出物与输出契约** —— 执行声明的交付物（报告、补丁、链接）。模板标记哪些产出物是*必需*的；只有所有必需产出物都已产出，执行才算**成功**。产出物可在执行详情的「产出物」页签下载。

**检查点（Checkpoint）** —— 模板定义的人工暂停点。等待期间不占用任何计算资源，任何项目成员都可以响应。共三种：**审批**（批准/驳回——在循环内还可*要求修改*，附上的意见会交给代理修订）、**问题表单**（代理提出结构化的澄清问题，每个都带可一键采纳的建议答案；若无问题则自动继续）、**评审关卡**（评审者的问题清单，勾选需要修复的项——未勾选的会被明确接受并列入 PR 描述——或全部接受）。检查点有超时时间，超时后按模板定义处理。

**循环（Loop）** —— 模板中可重复的段落，轮数有固定上限：澄清问答会循环到代理不再提问为止；评审-修复循环会持续到评审者不再报告问题、或你接受剩余问题为止。执行时间线会标注每一轮（「第 2/3 轮」）。

**评论（Comments）** —— 每次执行都有一条嵌在时间线中的讨论线程。成员可随时评论；评论对所有正在观看的人实时可见。

**预算与配额** —— *预算*属于模板/执行：单次执行的费用与时长上限（可细化到阶段）。*配额*属于项目：所有执行的每月费用/Token 上限。开启**强制停止**后，配额耗尽会拒绝新提交、并在下一个步骤边界中止进行中的执行；未开启则仅作提示。

**资源授权（Grant）** —— 让注册表中的资源（模型、技能、MCP 服务、硅基人）在某个项目内可用的开关。注册表是组织级的，授权是项目级的——管理员由此控制哪个团队能用哪个模型、哪个集成。

**执行状态** —— `排队中 → 执行中 → 已成功 | 已失败 | 已取消 | 已超时`，审批节点期间为 `执行中 ⇄ 待审批`。失败的执行带有机器可读的错误码与人类可读的说明。
